\chapter{Techniken}
Es existieren viele relevante Algorithmen mit polynomialer Laufzeit, welche auf vielen Daten angewandt werden müssen.
Diese Vorlesung behandelt verschiedene Techniken, welche verwendet werden können, um möglichst viele Daten möglichst schnell zu verarbeiten.
Dabei geht es hauptsächlich nicht um algorithmische Optimierungsschritte, sondern um die vollständige Ausnutzung der Computer Hardware.

Allgemeine Techniken, die in dieser Vorlesung vorgestellt werden sind: Cache- und Speicher-Optimierung, Vektorisierung, Parallelisierung, Verteiltes Rechnen und Rechnen auf Grafikkarten.

In diesem Kapitel werden die referenzierten Techniken erklärt.

\section{Speicher und Latenzen}
Der Hauptspeicher ist aufgeteilt in den logischen und physischen Speicher.
Dabei besitzt jeder Prozess einen logischen Adressraum, welcher indirekt aus dem physischen Speicher zugeordnet wird.

\paragraph{Pages}
Adressen im Hauptspeicher werden zu \emph{Pages} zusammengefasst, welche immer nur vollständig geladen werden können.
Wenn ein Prozess eine Page das erste mal benutzt, wird sie erst im physischen Speicher zugewiesen, dementsprechend können gewisse Verzögerungen entstehen beim initialen Aufruf.
Wenn physischer Speicher ausgeht, können Pages auf die Festplatte auslagern, was sehr langsam werden kann.

\paragraph{Cachelines und Cache}
Reihen von Pages werden als \emph{Cacheline} immer nur vollständig zusammen geladen.
Dementsprechend ist es schneller Daten nah beieinander im physischen Speicher zu speichern, sodass so wenig Cachelines direkt nacheinander geladen werden müssen wie möglich.

Beispielweise ist es schneller ein Reihen-verteiltes zweidimensionales Feld \lstinline|int vec[n * m]| an der Stelle \(i \in [n], j \in [m]\) mit Adressierung \lstinline|vec[i + j * n]| so zu indizieren, dass \(i\) schneller als \(j\) inkrementiert wird.

Cachelines werden im Cache zwischengespeichert.
Es gibt drei Caches in der CPU, den \emph{1st}, \emph{2nd}, und, \emph{3rd Level Cache}, die sich unterscheiden in Größe und Geschwindigkeit.
Dabei ist der 1st Level Cache der schnellste und kleinste mit beispielweise 32 KB pro Kern, währenddessen der 3rd Level Cache am langsamsten und größten ist mit beispielweise 32 MB geteiltem Speicher.

\paragraph{Thrashing}
Da der Cache im Vergleich zum physischem Speicher sehr klein ist, wird der gleiche Bereich im Cache oft von verschiedenen Adressen im Hauptspeicher verwendet.
Dementsprechend, wenn zwei Cachelines den gleichen Bereich im Cache zugeordnet werden, kann es zu \emph{Trashing} kommen, wenn beide gleichzeitig oder nacheinander verwendet werden müssen.
Da nur eine Cacheline gleichzeitig in der Stelle im Cache liegen kann, wird dann wiederholt die derzeitige Cacheline verworfen, und die andere Cacheline aus dem Hauptspeicher gelesen.

Um Trashing zu verhindern, kann es oftmals praktisch sein, die Problemgröße anzupassen und keine Zweierpotenzen zu verwenden beim Zuordnen von Speicher.
Somit wird verhindert, dass zwei Variablen so gespeichert werden, dass ihre Cachelines überlappen.

\paragraph{Superskalare Berechnungen}
Wenn ein Programm Daten aus dem Hauptspeicher lädt, kann das mehrere Hundert Taktzyklen dauern.
Damit die CPU in dieser Zeit nicht blockiert, können Befehle so geordnet werden, dass während dem Warten etwas anderes berechnet wird.
Dies ist offensichtlich nur möglich, wenn die Befehle unabhängig sind.

Eine Technik die dies zu nützen macht, ist \emph{Loop unrolling}.
Hierbei werden parallelisierbare Schleifen so aufgeteilt, dass beim Warten eine anderer Befehl berechnet werden kann, siehe \autoref{lst:1}.

\begin{lstlisting}[label={lst:1}, caption={Loop unrolling über die cumulative Sinus-Funktion.}]
// Naive
for (int i=0; i<n; i++) {
    x[i] = sin(y[i]);
}

// Loop unrolling
for (int i=0; i+1 < n; i+=2) {
    x[i] = sin(y[i]);
    x[i+1] = sin(y[i+1]);
}
\end{lstlisting}

\pagebreak

Eine alternative Technik ist das sog. \emph{Prefetching}, wo Daten vorzeitig in den Cache geladen werden, bevor sie verwendet werden müssen, siehe \autoref{lst:2}.
Der Vorteil dieser Technik ist, dass so auch Daten, von denen der Compiler noch nicht einschätzen kann, wann sie geladen werden müssen, geladen werden können.
Dies wäre zum Beispiel der Fall bei \lstinline|x[i] = z[y[i]]|, wo die Position in \(z\) abhängig vom Wert in \(y\) ist, und somit laufzeitabhängig sein kann.

\begin{lstlisting}[label={lst:2}, caption={Prefetching für \autoref{lst:1}.}]
// Prefetching
for (int i=0; i<n-50; i++) {
    __builtin_prefetch(&y[i+50]);
    x[i] = sin(y[i]);
}

for (int i=n-50; i < n; i++) {
    x[i] = sin(y[i]);
}
\end{lstlisting}

\paragraph{Streaming}
Wenn ein Programm nur schreiben will, können unnötige Latenzen entstehen, da die jeweilige Cacheline erst gelesen werden muss, und somit in den Cache geladen wird.
Alternativ können diese Latenzen und das unnötige Füllen vom Cache umgangen werden mit sog. \emph{Streaming}, wo Schreibzugriffe am Cache vorbei durchgeführt werden.

Dies ist zum Beispiel möglich im AVX2 Standard\footnote{Mehr hierzu in \autoref{sec:2}.}, wo 256 Bit Vektoren, also 8 \lstinline|float| Werte am Laufband vom Feld \lstinline|float y[8]| in das Feld \lstinline|float x[8]| geschrieben werden können mittels \lstinline|_mm256_stream_ps(x, _mm256_load_ps(y));|

\paragraph{Pipelining}
Die Idee hinter \emph{Pipelining} ist es, Latenzen zu verstecken, indem unabhängige Operationen gleichzeitig ausgeführt werden, wie beim Loop unrolling und Prefetching.

Ein Beispiel hinter dieser Technik ist das Horner-Schema.
Ein Polynom \(a_0, \dots, a_n\) kann mit schlauer mit dem Horner-Schema an der Stelle \(x\) ausgewertet werden mit
\[(((a_n x + a_{n-1}) x + a_{n-2}) \dots ) x + a_0.\]
Offenbar muss so der ganze Term von links berechnet werden.
Alternativ lassen sich jedoch die Potenzen aufteilen in unabhängige gerade und ungerade, die gleichzeitig berechnet werden können.

\section{Vektorisierung}\label{sec:2}
Vektorisierung ist eine weitere Technik, die verwendet werden kann, um mehrere Daten gleichzeitig zu berechnen.
Generell verläuft Vektorisierung nach dem SIMD (Single Instruction, Multiple Data) Prinzip.
Wir wenden also die gleiche Operation auf mehrere Daten gleichzeitig an.
Dabei werden nach AVX und AVX2 spezielle Vektorregister verwendet, die jeweils 128 und 256 Bit gleichzeitig berechnen können.
In AVX kann mit u.a., Acht 32 Bit \lstinline|float| Werten gleichzeitig gerechnet werden, in AVX2 sind es Sechzehn.

AVX Register können explizit über Intrinsics verwendet werden, es kann aber auch automatisch durch den Compiler vektorisiert werden\footnote{\texttt{-ffast-math} muss als Compiler Flagge gesetzt sein, damit Gleitkommazahlen vektorisiert werden dürfen.}.

Allgemein wird vektorisiert wie in \autoref{lst:3}.
In diesem Beispiel müssen noch die Acht Teilsummen zusammen und die restlichen Werte\footnote{Wenn \(8 \not\mid n\).} addiert werden.
Die Gängigen AVX2 Typen und Operationen sind vorzufinden in \autoref{lst:4}.

\begin{lstlisting}[label={lst:3}, caption={Explizite Vektorisierung mit AVX2.}]
// Ohne Vektorisierung
float sum = 0.0f;
for (int i=0; i<n; i++)
    sum += x[i]

// Mit Vektorisierung
__m256 v_sum = _mm256_set1_ps(0.0f);
for (int i=0; i<n-7; i+=8;)
    sum = _mm256_add_ps(sum, _mm256_loadu_ps(&x[i]);
\end{lstlisting}

\begin{lstlisting}[label={lst:4}, caption={Wichtige AVX2 Typen und Operationen auf 8 \lstinline|float| Werten.}]
// 8 floats
__m256
// 4 doubles
__m256d
// 8 ints
__m256i

// Lesen und Schreiben auf durch 32 teilbare Adressen
_mm256_load_ps, _mm256_store_ps
// Lesen und Schreiben auf beliebige Adressen
_mm256_loadu_ps, _mm256_storeu_ps
// Maskiertes Lesen und Schreiben
_mm256_maskload_ps, _mm256_maskstore_ps
// Streaming
_mm256_stream_ps
\end{lstlisting}

\paragraph{Structure of Arrays}
Möglicherweise müssen verwendete Datenstrukturen angepasst werden, damit sie effizient Vektorisiert werden können.
Beispielweise speichert man dafür lieber Felder von Positionen und Massen, als ein Feld von einzelnen Positionen und Massen.

\paragraph{Fallunterscheidungen}
Etwas anstrengend wird es, wenn Fallunterscheidungen nötig sind.
Dabei braucht konventionell möglicherweise jeder einzelne Skalar im Vektor einen eigenen Pfad.
Jedoch gibt es auch hierfür einen netten Trick um die naive Lösung zu umgehen.
Man kann beide Pfade explizit berechnen, sodass man für \lstinline|float| Typen beispielweise zwei \lstinline|__m256| Vektoren hat, die den ersten und zweiten Fall repräsentieren.
Dann erstellt man eine Bitmaske mit \lstinline|_mm256_cmp_ps|\footnote{Oder \lstinline|_mm256_cmp_ps_mask| in AVX-512.}, den Vergleich der Fallunterscheidung ausführt.
Schließlich führt man anhand dieser Bitmaske die Ergebnisse zusammen mit \lstinline|_mm256_blendv_ps|.

Alternativ lassen sich auch andere Tricks ausnutzen, um eine Fallunterscheidung komplett zu umgehen.
Beispielweise kann man ein Vorzeichen auch direkt als Maske speichern und dann clever beide Pfade zusammenführen in einen, der aber nun abhängig von dieser Maske vorgeht.

\paragraph{Newton-Iteration}
Einige Operationen im AVX2 Satz sind nur Approximationen für gewisse Funktionen und teilweise nicht genau genug.
Es lässt sich jedoch die Genauigkeit verbessern, indem wir eine Newton-Iteration dazurechnen.
Dies lässt sich beispielweise bei der reziproken Wurzel anwenden.

\paragraph{Permutationen}
Offenbar will man Komponenten in einem Vektor umsortieren.
Das kann man mit verschiedenen Methoden machen, siehe \autoref{lst:5}.

\begin{lstlisting}[label={lst:5}, caption={Umsortieren in einem \lstinline|__m256|}.]
__m256 a,b,c;
// $m$ binäre Zahl bestehend aus 2er Ziffern $\alpha, \beta, \gamma, \delta$
int m;
__m256i i;

// $c \leftarrow [a_\alpha, a_\beta, b_\gamma, b_\delta, a_{\alpha + 4}, a_{\beta + 4}, b_{\gamma + 4}, b_{\delta + 4}]$
c = _mm256_shuffle_ps(a, b, m);
// $c \leftarrow [ab_{\alpha..\alpha+4}, ab_{\beta..\beta+4}, \dots]$
c = _mm256_permute2f128_ps(a, b, m);
// $c \leftarrow [a_1, b_1, a_2, b_2, a_5, b_5, a_6, b_6]$
c = _mm256_unpacklo_ps(a, b);
// $c \leftarrow [a_3, b_3, a_4, b_4, a_7, b_7, a_8, b_8]$
c = _mm256_unpackhi_ps(a, b);
// $c \leftarrow [a_{i_0}, a_{i_1}, \dots, a_{i_8}]$
c = _mm256_permutevar_ps(a, i);
\end{lstlisting}

\paragraph{Generelle Operationen}
Operationen werden mit den \lstinline|_mm256_$a$_$b$| Intrinsics ausgeführt, wobei \(a \in \{\text{\lstinline|add|}, \text{\lstinline|sub|}, \text{\lstinline|mul|}, \text{\lstinline|div|}\}\) für \(b \in \{\text{\lstinline|pd|}, \text{\lstinline|ps|}, \text{\lstinline|epi8|}, \text{\lstinline|epi16|}, \dots, \text{\lstinline|epi64|}\}\). Für die ganzzahligen Typen gibt es auch \(a \in \{\text{\lstinline|and|}, \text{\lstinline|or|}, \text{\lstinline|xor|}\}\) sowie Bit Shifts und Fallunterscheidungen.

Da es keine Bit Shifts für Gleitkommazahlen gibt, kann es oft schlau sein eine Gleitkommazahl zu einer Ganzzahl hin und zurück zu konvertieren, um Bitmasken anzuwenden.

\section{Multithreading mit OpenMP}
Es gibt verschiedene Arten der Parallelisierung, erst beschäftigen wir uns mit Multithreading welches mit mehreren Prozessorkernen gleichzeitig rechnet.
Dafür verwenden wir den OpenMP (OMP) 4.0 Standard.

\paragraph{Threads} \emph{Threads} werden von Kernen ausgeführt.
Sie haben einen eigenen Befehlszähler, eigene Register, und einen eigenen Stack.
Threads werden einem Prozess zugeordnet, und teilen sich jeweils den assoziierten Adressraum.

OMP verwendet \lstinline|#pragma|-Direktiven, um Teams von Threads zu spezifizieren, siehe \autoref{lst:6}.
Die Nummer eines Threads sowie die Nummer an erzeugten Threads wird von jedem Thread berechnet und ausgegeben.
Der Thread mit Nummer 0 wird oft als \emph{Anführer} verwendet.

\begin{lstlisting}[label={lst:6}]
#pragma omp parallel
{
    int rank = omp_get_thread_num();
    int size = omp_get_num_threads();

    printf("Thread %d/%d reporting for duty\n", rank, size);
}
\end{lstlisting}

In OMP gibt es private und geteilte Variablen zwischen Threads.
Variablen, die innerhalb eines parallelisierten Bereiches deklariert werden, sind immer privat, währenddessen äußere Variablen normalerweise geteilt werden.
Äußere Variablen können aber auch privat gesetzt werden, indem im pragma zusätzlich noch \lstinline|firstprivate($\dots$)| spezifiziert wird.

Es kann außerdem \lstinline|#pragma omp for| in einem parallelisierten Bereich direkt vor eine einfache \lstinline|for|-Schleife gesetzt werden, damit die Threads automatisch auf die Iterationen geteilt werden -- das kann aber auch manuell über \lstinline|omp_get_thread_num| und \lstinline|omp_get_num_threads| implementiert werden.
Wenn direkt nacheinander \(n\) \lstinline|for|-Schleifen nach einem \lstinline|#pragma omp for| kommen, kann \lstinline|collapse($n$)| hinzugefügt werden, um die \lstinline|for|-Schleifen zusammenzufassen.

\paragraph{Barrieren}
Oft müssen Threads an einer Stelle synchronisiert werden um Wettlaufsituationen (eng. \emph{Race Conditions}) zu verhindern.
Mit \lstinline|#pragma omp barrier| wird garantiert, dass alle Threads die jeweilige Zeile erst erreichen müssen, bevor sie weiter dürfen.
Eine Barriere wird implizit auch nach einer parallelisierten \lstinline|for|-Schleife eingefügt.

\paragraph{Speicheraktualisierung}
Da sich Threads nicht Register teilen, muss manchmal garantiert werden, dass ein Wert einer geteilten Variable vollständig aktualisiert wurde.
Das geht mit \lstinline|#pragma omp flush|.

\paragraph{False Sharing}
Wenn mehrere Threads gleichzeitig auf die gleiche Cacheline zugreifen, kann es sein, dass weitere Latenzen vorkommen, da die Cachelines erst synchronisiert werden müssen.
Sobald ein Thread eine Cacheline verändert, wird diese für die anderen Threads invalidiert und muss aktualisiert werden.
Das nicht nur der Fall wenn die gleiche geteilte Variable verändert wird, sondern auch, wenn nah beinahe private Variablen, verändert werden.

\subsection{Nichtdeterministisches Verhalten}
Wenn mehrere Threads auf die gleiche Variable schreiben, kann es zu Wettlaufsituationen kommen, wodurch der Wert der Variable nicht deterministisch wird.
Das liegt offenbar daran, dass einige Threads die Variable lesen, bevor sie verändert wird von einem anderen Thread, aber danach auf sie schreiben.

\paragraph{Atomare Operationen}
Solche Situationen können durch u.a. \emph{atomare Operationen} verhindert werden.
Mit \lstinline|#pragma omp atomic| wird garantiert, dass die folgende Operation, wie zum Beispiel \lstinline|i++|, unteilbar ausgeführt wird, was von der Hardware garantiert werden kann.

\paragraph{Kritische Abschnitte}
Ähnlich wie atomare Operationen kann man auch einen generellen Bereich mit \lstinline|#pragma omp critical| versehen wird, der die jeweilige Zeile für andere Threads blockiert, wenn sie erreicht wird.
Das ist allerdings ziemlich langsam, weil dann immer alle Threads synchronisiert werden müssen an der Stelle.

\paragraph{Reduktion}
Eine weitere Methode ist die \emph{Reduktion} einer Variable auf die gleichzeitig geschrieben werden soll.
Hierfür schreibt man an den \lstinline|#pragma omp parallel| Befehl noch den Parameter \lstinline|reduction($id$:$v, \dots$)|, wobei \(id\) spezifiziert wie reduziert wird, z.B. \lstinline|+|, \lstinline|-|, \lstinline|*|, \lstinline|&&|, \lstinline&||&, \lstinline|max|, \lstinline|min|, und \(v, \dots\) meint, welche Variablen reduziert werden sollen.
Das funktioniert, indem temporär pro Thread eine private Version der Variable modifiziert wird, die nach dem Block wieder zusammen gefügt wird.

\paragraph{Locks}
\emph{Locks} sind eine Generalisierung von kritischen Abschnitten, die verwendet werden können um zu kontrollieren, wann gleichzeitig bestimmte Regionen ausgeführt werden sollen, und wann nicht.
Locks werden mit dem Datentyp \lstinline|omp_lock_t| repräsentiert.
Durch \lstinline|omp_init_lock| und \lstinline|omp_destroy_lock| werden sie angelegt und gelöscht.
Ein nicht geschlossenes Lock kann mit \lstinline|omp_set_lock| abgeschlossen werden.
Erreicht nun ein Thread ein abgeschlossenes Lock und versucht es aufzuschließen, blockiert dieser Thread, bis es wieder aufgeschlossen wurde, und somit wieder abgeschlossen werden kann.
Ein Lock wird mit \lstinline|omp_unset_lock| aufgeschlossen.
Um herauszufinden ob ein Lock geschlossen oder aufgeschlossen ist, kann \lstinline|omp_test_lock| verwendet werden.

\paragraph{Tasks}
In OMP kann man Abschnitte als \emph{Tasks} markieren.
Wenn ein Team ein Task erreicht, führt ein Thread des Teams diese Aufgabe aus.
Ein Task kann eine Zeile oder ein Block sein und wird mit \lstinline|#pragma omp task| markiert.
Tasks können verwendet werden, um unabhängige Aufgaben zu erledigen.
Mit \lstinline|#pragma omp taskwait| kann ein Team warten bis alle Tasks im Block abgeschlossen wurden.

Offenbar will man nicht, dass immer alle Threads warten müssen, wenn synchronisiert werden muss.
In OMP 4.0 wurden deswegen Task Abhängigkeiten eingeführt, mit denen spezifiziert werden kann, welche Tasks von anderen abhängig sind.
Abhängigkeiten werden mit der \lstinline|depend($t$: $v, \dots$)|-Klausel in einem Task deklariert.
Dabei kann \(t\) entweder \lstinline|in|, \lstinline|out|, oder \lstinline|inout| sein, je nachdem wie die Abhängigkeit über Variablen \(v, \dots\) funktionieren soll.
Tasks werden dann so ausgeführt, dass alle Tasks mit einer \lstinline|in|- oder \lstinline|out|-Abhängigkeit für \(x\) auf alle vorherigen Tasks warten, die auf \(x\) schreiben (\lstinline|out| und \lstinline|inout|).
Tasks mit einer \lstinline|inout|-Abhängigkeit für \(x\) warten auf alle vorherigen Tasks, die \(x\) lesen oder auf \(x\) schreiben (\lstinline|in|, \lstinline|out|, \lstinline|inout|).

Schleifen können auch in Tasks aufgeteilt werden mit \lstinline|#pragma omp taskloop|, wo ein bestehendes Team von Threads automatisch auf einzelne Iterationen aufgeteilt wird.

\paragraph{SIMD in OpenMP}
OMP kann auch verwendet werden, um automatisch \lstinline|for|-Schleifen zu vektorisieren mit \lstinline|#pragma omp simd|.
Somit kann effizient gleichzeitig parallelisiert und automatisch vektorisiert werden.

\section{Schnelle Verfahren für Vielkörpersysteme}
Je nach Problem können auch Approximationsalgorithmen verwendet werden, die schnell und genau sind.
In der Vorlesung wird das am Beispiel der Berechnung von Gravitationskräften diskutiert.
Hierfür ist die Idee den Raum hierarchisch zu partitionieren durch Clustering und die Gravitationskräfte durch Tensor-Interpolation darzustellen.
Somit kann die Kraft zwischen entfernten Sternen approximiert werden, indem einem Sterncluster ein Gravitationspotential interpoliert wird.
Dabei kann beim symmetrischen Verfahren die Approximation verwendet werden, wenn der Abstand zwischen Clustern größer ist, als die jeweiligen Diameter.

Das ganze erfolgt beispielweise symmetrisch über Vorwärts- und Rückwärtstransformationen bei denen jeweils rekursiv die Gravitationsfaktoren für alle Cluster berechnet werden und berechnete Kräfte zu Vaterknoten weitergegeben werden.

\section{Verteiltes Rechnen mit MPI}
Die Idee hinter verteiltem Rechnen ist eine Parallelisierung zwischen verschiedenen Geräten die erfolgt indem Nachrichten in einem Netzwerk ausgetauscht werden zwischen einzelnen Systemen.
Der Message Passing Interface (MPI) Standard wird oft verwendet, um ein Problem verteilt zu lösen und die nötige Kommunikation zu regeln.

MPI wird über eine Bibliothek in C Code eingebunden, der mit einem seperaten Compiler namens \emph{mpicc} übersetzt wird, und mit \emph{mpirun} ausgeführt werden muss.

\paragraph{Prozesse}
Die Parallelisierung erfolgt über Prozesse und nicht über Threads wie in OMP.
Dazu ist zu beachten, dass Prozesse ihren eigenen Adressraum besitzen, somit gibt es also keine direkt geteilten Variablen zwischen Prozessen.

\paragraph{Kommunikatoren}
In MPI wird über spezielle \emph{Kommunikatoren} mit Typ \lstinline|MPI_Comm| \emph{kommuniziert}\footnote{;)}.
Sie ordnen jedem Prozess einen Rang zu und stellen sicher, dass synchrone Nachrichten in der Reihenfolge empfangen werden, in der sie gesendet werden.
MPI definiert automatisch die Kommunikatoren \lstinline|MPI_COMM_WORLD| und \lstinline|MPI_COMM_SELF|, jedoch können noch weitere zur Laufzeit erstellt werden.

Nachrichten besitzen einen Absender und Empfänger, welche durch den Rang zugeordnet werden.
Außerdem enthalten Nachrichten eine Markierung und eine Anzahl von Elementen eines Typs.
Ein simples Beispielprogramm, das MPI einbindet und zwischen zwei Prozessen eine Nachricht austauscht, ist zu finden in \autoref{lst:7}.

\begin{lstlisting}[label={lst:7}, caption={Ein einfaches MPI Programm}]
int main(int argc, char** argv) {
    char buf[6];
    int rank;

    MPI_Init(&argc, &argv);
    MPI_Comm_rank(MPI_COMM_WORLD, &rank);

    if (rank == 0)
        MPI_Send("Hello", 6, MPI_CHAR, 1, 0, MPI_COMM_WORLD);
    else if (rank == 1) {
        MPI_Recv(buf, 6, MPI_CHAR, 0, 0, MPI_COMM_WORLD,
        MPI_STATUS_IGNORE);
        printf("%s World\n", buf);
    }

    MPI_Finalize();
    return 0;
}
\end{lstlisting}

Die wichtigsten Methoden in MPI werden in \autoref{lst:8} erklärt.

\begin{lstlisting}[label={lst:8}, caption={Grundlegende Funktionen in MPI.}]
// Startet MPI
int MPI_Init(int* argc, char** argv);
// Beendet MPI
int MPI_Finalize();

// Ermittelt die Anzahl an Prozessen
int MPI_Comm_size(MPI_Comm comm, int* rank);
// Ermittelt den Rang des aktuellen Prozesses
int MPI_Comm_rank(MPI_COMM comm, int* size);

// Sendet buf an dest
int MPI_Send(void* buf, int count, MPI_Datatype type, int dest, int tag, MPI_Comm comm);
// Empfängt eine Nachricht von src
int MPI_Recv(void* buf, int count, MPI_Datatype type, int src, int tag, MPI_Comm comm, MPI_Status* status);
\end{lstlisting}

\paragraph{Tags}
\emph{Tags} werden beim Senden und Empfangen einer Nachricht verwendet.
Die Tags von Sende- und Empfangsoperationen müssen gleich sein, wobei \lstinline|MPI_ANY_TAG| beim Empfangen zu allen Tags passt.

\paragraph{Synchrones und Asynchrones Senden}
\lstinline|MPI_Send| und \lstinline|MPI_Recv| können warten, bis eine Empfangsoperation begonnen hat oder abgeschlossen wurde, was unter Umständen zu Deadlocks führen kann.
Wenn \lstinline|MPI_Send| warten soll, bis eine Empfangsoperation begonnen hat, kann man \lstinline|MPI_Ssend| verwenden.
Alternativ lassen sich \lstinline|MPI_Isend| und \lstinline|MPI_Irecv| verwenden, die nicht blockieren.
Bei diesen Funktionen wird ein \lstinline|MPI_Request| Pointer noch hinzugegeben, welcher von \lstinline|MPI_Wait| verwendet werden kann, um bis zu der jeweiligen Fertigstellung zu warten.

\paragraph{Kollektive Kommunikation}
Einige Operationen können effizient ausgeführt werden, wenn mehrere Prozesse direkt zusammen arbeiten.
Methoden, die die nötige Kommunikation zwischen allen Prozessen realisieren sind in \autoref{lst:9} zu finden.

\begin{lstlisting}[label={lst:9}, caption={Kollektive Kommunikation in MPI.}]
// Sendet an alle anderen Prozesse eine Nachricht
int MPI_Bcast(void* buf, int count, MPI_Datatype type, int root, MPI_Comm comm);

// Sammelt Daten von allen Prozessen und reduziert für root, verschickt dabei jeweils sendbuf
int MPI_Reduce(void* sendbuf, void* recvbuf, int count, MPI_Datatype type, MPI_Op op, int root, MPI_Comm comm);
\end{lstlisting}

Dabei wird der Parameter \lstinline|root| verwendet, um die Rang der Quelle zu spezifizieren, und \lstinline|op|, wie die Daten zusammengeführt werden sollen bei der Reduktion.
Beispielweise können \lstinline|MPI_MAX|, \lstinline|MPI_MIN|, \lstinline|MPI_SUM|, oder \lstinline|MPI_PROD| für \lstinline|op| verwendet werden.

Es gibt außerdem nicht-blockierende Versionen von \lstinline|MPI_Bcast| und \lstinline|MPI_Reduce|, sowohl eine Funktion \lstinline|MPI_Allreduce|, die zuerst reduziert, um dann das Ergebnis an alle anderen Prozesse zu schicken.

\paragraph{Scatter und Gather}
\emph{Scatter}- und \emph{Gather}-Operationen sind spezielle Formen von kollektiver Kommunikation, bei denen jeweils einzelne Elemente eines Arrays an verschiedene Prozesse geschickt werden, sowie ein Array aus einzelnen empfangenen Elementen verschiedener Prozesse zusammengebaut wird.
Hierfür werden die \lstinline|MPI_Scatter| und \lstinline|MPI_Gather| Funktionen verwendet.
Für Arrays mit Elementen unterschiedlicher Länge existieren außerdem noch \lstinline|MPI_Scatterv| und \lstinline|MPI_Gatherv|.

\paragraph{All-to-all}
Um Nachrichten von allen Prozessen zu allen anderen zu schicken, kann auch statt einem Aufruf von \lstinline|MPI_Bcast| bei jedem Prozess, die Funktion \lstinline|MPI_Alltoall| verwendet werden, welche dann zusätzlich die eingehenden Nachrichten automatisch annimmt.
Von dieser Funktion existiert wieder eine Version mit variabler Nachrichtlänge.

\paragraph{Einseitige Kommunikation}
In MPI werden außerdem Methoden realisiert, die eine einseitige Kommunikation möglich machen, bei der ein Prozess auf Daten anderer direkt zugreifen kann.
Das erfolgt über sog. Fenster, mit denen ein Prozess einen Teil seines Speichers an andere anbieten kann.

\section{Rechnen auf Grafikkarten}
Das Rechnen auf GPUs kann für einige parallelisierbare Probleme erheblich schneller funktionieren als auf CPUs.
GPUs sind gut dazu geeignet gleichzeitig die gleiche Operationen für viele Daten anzuwenden, aber schlecht für Probleme mit komplexen Fallunterscheidungen.
GPUs können schneller auf den Grafikspeicher zugreifen als CPUs auf den Hauptspeicher, jedoch gibt es weniger Grafikspeicher und es muss koordiniert werden, wie effizient Daten vom Hauptspeicher auf den Grafikspeicher kopiert werden.

Generell verläuft GPU Programmierung nach dem SIMT (Single Instruction Multiple Threads) Prinzip.
Ein Programm erstellt sehr viele GPU Threads, welche nur aus einem Befehlszähler und wenigen Registern bestehen, aber keinem Stack!

\paragraph{Threadblöcke}
Threads werden in \emph{Threadblöcken} zusammengefasst.
Threads im gleichen Threadblock teilen sich Register und können sich synchronisieren.

\paragraph{Warps}
Threadblöcke werden in \emph{Warps} à la typischerweise 32 Threads zerlegt.
In jedem Taktzyklus wählt die GPU einen Warp und eine Operation, die für alle Threads im Warp ausgeführt wird, wenn sie ausgeführt werden muss.

\subsection{CUDA}
NVIDIAs Compute Unified Device Architecture (CUDA) ist eine Erweiterung von C++, die es ermöglicht Maschinencode für CPUs, GPUs, sowie deren Koordination zu schreiben.
Leider läuft CUDA nur auf NVIDIA Grafikkarten, ist jedoch sehr leicht zu lernen.
In CUDA werden einige Begriffe eingeführt.

\begin{description}
\item[Host] Computer, der die Grafikkarte initialisiert, mit Daten versorgt, und CUDA Threads startet.
\item[CUDA Device] Gerät mit Multiprozessoren und Grafikspeicher, das verbunden ist mit dem Host.
\item[Multiprozessor] Besteht aus Kernen, Scheduler, kleinem gemeinsam genutzten Speicher, und großem Registersatz.
\item[Kerne] Führen die Operationen von Threads aus.
\end{description}

\paragraph{CUDA Kernel}
In der CUDA Sprache werden Funktionen, die auf der GPU ausgeführt werden sollen als Kernel mit \lstinline|__global__| markiert.
Pointer in solcher Methode beziehen sich stehts auf den Grafikspeicher.
Mit \lstinline|blockIdx.x * blockDim.x + threadIdx.x| lässt sich ein Rang für den jeweiligen Thread berechnen.

Bevor ein Kernel aufgerufen werden kann, muss der nötige Grafikspeicher angefordert werden.
Das erfolgt mit \lstinline|cudaMalloc(void** mem, size_t size)|.
Der Transfer zwischen CPU Pointer und GPU Pointer wird mit \lstinline|cudaMemcpy(void* dest, void* src, size_t size, enum kind)| realisiert.
Der Parameter \lstinline|kind| kann dabei entweder \lstinline|cudaMemcpyHostToDevice| oder \lstinline|cudaMemcpyDeviceToHost| sein.

Ein Kernel \lstinline|func| wird mit \lstinline|func<<<($T$, $B$>>>($\dots$)| aufgerufen, wobei \(B\) die Blockgröße ist, und \(T\) die Anzahl an Threadblöcken darstellt.
Typischerweise haben wir \(B = 256\) und \(T = (n+255)/256\) für Problemgröße \(n\).

Nach dem Aufruf vom Kernel rechnet die CPU nebenbei weiter, kann aber mit \lstinline|cudaDeviceSynchronize| auf das Ende vom Kernel warten.

\paragraph{Unified Memory}
Alternativ zum manuellen Übertragen zwischen Daten im Hauptspeicher und Grafikspeicher, kann CUDA den Transfer automatisch organisieren.
Das funktioniert mit \lstinline|cudaMallocManaged(void** mem, size_t size)|.
Der Pointer \lstinline|mem*| wird nun automatisch im Kernel als Grafikadresse interpretiert, und der Transfer zwischen GPU und CPU Speicher funktioniert intime ähnlich wie Cache.

\paragraph{Streams}
Eine schlauere Methode den Datentransfer zu regeln erfolgt durch \emph{Streams} -- einem nicht-blockierenden Datentransfer.
Die Idee ist, gleichzeitig auf der GPU zu rechnen während auf die nächsten Daten gewartet wird.
Ein Beispielprogramm, das Streams verwendet ist in \autoref{lst:10} zu finden.

\begin{lstlisting}[label={lst:10}, caption={Ausführen eines Kernels mit 2 Streams.}]
#define STREAMS 2

float* dev_x[STREAMS];
float* dev_y[STREAMS];

cudaStream_x streams[STREAMS];

for (int i = 0; i < STREAMS; i++) {
    // Erstelle alle Streams
    cudaStreamCreate(&streams[i]);
    // Erstelle Blöcke mit Größe n / STREAMS
    cudaMalloc(&dev_x[i], n / STREAMS * sizeof(float));
    cudaMalloc(&dev_y[i], n / STREAMS * sizeof(float));
}

for (int i = 0; i < STREAMS; i++) {
    // Fülle alle Streams mit Daten.
    cudaMemcpyAsync(dev_x[i], &x[i * (n / STREAMS)], sizeof(float) * (n / STREAMS), cudaMemcpyHostToDevice, streams[i]);
    // Führe den Kernel mit dem aktuellen Stream aus.
    kernel<<<(n/STREAMS + 255)/256, 256, 0, streams[i]>>>(n / STREAMS, dev_x[i], dev_y[i]);
    // Kopiere die Daten wieder in den Hauptspeicher.
    cudaMemcpyAsync(&y[i * (n / STREAMS)], dev_y[i], sizeof(float) * (n / STREAMS), cudaMemcpyDeviceToHost, streams[i]);
}

// Synchronisiere Streams und zerstöre sie.
for (int i = 0; i < STREAMS; i++) {
    cudaStreamSynchronize(streams[i]);
    cudaFree(dev_x[i]);
    cudaFree(dev_y[i]);
    cudaStreamDestroy(streams[i]);
}
\end{lstlisting}

\subsection{OpenCL}
Eine Alternative zu CUDA, die auf vielen Geräten funktioniert, ist OpenGL.
Leider ist OpenCL durchaus komplexer als CUDA und hat eine etwas lästige Designphilosophie.
Um ein Programm in OpenCL zu starten, müssen die folgenden Schritte erfolgen.

\begin{enumerate}
    \item Finde und wähle die Plattform (Verschiedene Implementierungen von OpenCL, z.B. NVIDIA CUDA, rusticl).
    \item Finde und wähle das Gerät oder die Geräte (GPU0, GPU1, CPU).
    \item Erstelle einen Kontext, der Geräte der gleichen Plattform koordinieren kann.
    \item Erstelle ein OpenCL Programm für den Kontext durch einen Quelltext als String.
    \item Übersetze das Programm in Maschinencode für die Geräte.
    \item Erstelle den Kernel aus dem Programm.
    \item Lege Speicher für die Parameter (Pointer zu Input und Output) an.
    \item Binde alle erstellten Speicher einzeln zu den jeweiligen Parameter.
    \item Erstelle eine Warteschlange für Befehle, die nicht-blockierend ausgeführt werden.
    \item Füge an die Warteschlange einen Befehl, der die Inputs auf den GPU Speicher schreibt.
    \item Füge an die Warteschlange einen Befehl, der den Kernel ausführt.
    \item Füge an die Warteschlange einen Befehl, der die Ergebnisse in den Hauptspeicher schreibt.
    \item Zuletzt, warte bis alle Befehle in der Warteschlange ausgeführt wurden.
\end{enumerate}

Unterschiedlich zu CUDA, werden Threads tatsächlich \emph{Work Iterms} genannt, und Threadblöcke \emph{Work Groups}.
